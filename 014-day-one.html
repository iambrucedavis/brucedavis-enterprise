02 — The Mechanism structured output, not free text

The model returns a UI spec.

Claude is forced to call one tool. Its argument is the UI spec — an ordered, typed list of blocks. This is what comes back from the request "what does my first week look like?"

render_onboarding_ui · tool call [ STRUCTURED OUTPUT ]
{
"intent": "First-week schedule",
"summary": "Here's how your week is shaped…",
"blocks": [
{ "type": "schedule_timeline", "props": { … } },
{ "type": "progress_ring", "props": { … } }
]
}

Tool-use forcing guarantees the envelope. A Zod discriminated union validates every block's props before a single component mounts — an invalid block never reaches the screen. The renderer switches on type and mounts the matching React component.

03 — Generative UI the data drives the interface

The UI is generative.

No screen in this app is hardcoded. Every interface a visitor sees is rendered from a structured spec — change the request, and the UI changes with it. That is the technical claim the project demonstrates.

Generative, not hardcoded
The screen is a render of the model's JSON. New request, new blocks, new UI — with no interface code changed.
Schema-validated
Tool-use forces the shape; a Zod discriminated union enforces the contract. The envelope and the props are both gated.
Two demo modes
Five precomputed prompts render instantly with no key; bring an Anthropic key to generate UI live from any request.
In review — what I'd do differently
  • The five sample prompts are hand-authored UI specs — real shape, curated content — so the demo never needs an API key or spends a cent.
  • The tool's JSON Schema is deliberately permissive on block props; Zod does the real validation after the call. It's the reliable split for deep typed unions.
  • Six block types ship today. The registry is built to grow — a seventh component is a schema entry, a line in the prompt, and one file.
